TKU Deployment Guide — Complete Step-by-Step
How Your Automation Works (Read This First)
Your project is already wired for full automation. Here is exactly what happens after a git push:


Your VS Code
    │
    │  git push origin main
    ▼
GitHub Repository (d:\TKU)
    │
    ├── backend/** changed?
    │       └── GitHub Actions (.github/workflows/deploy-backend.yml)
    │               └── Pushes backend/ folder to HF Spaces git repo
    │                       └── HF Spaces rebuilds Docker container
    │                               └── start.sh runs → migrate.py runs → server starts
    │
    └── frontend/** changed?
            └── Vercel detects push automatically
                    └── Builds React app → deploys to your Vercel URL
You never need to touch HF Spaces or Vercel manually for routine updates. One git push does everything.

Part 2 — Hugging Face: How Backend Deploys
How it works
Your repository already has .github/workflows/deploy-backend.yml. Here is exactly what it does:

When you push to main AND any file inside backend/ changed — the workflow runs
It clones your HF Space's git repo (https://huggingface.co/spaces/YOUR-USERNAME/tku-backend)
It copies all backend/ files into the Space (excluding .env, tku.db, venv/)
It pushes to the HF Space repo
HF Spaces detects the new commit and rebuilds the Docker container automatically
start.sh runs inside the container:
Checks if /data/tku.db exists — it does, so no re-seeding (your data is safe)
Starts uvicorn
main.py runs migrate.py at startup — the new supplier_id column is added to the products table automatically
Step 2.1 — Watch the GitHub Actions deployment
Go to your GitHub repo → click the Actions tab
You will see a workflow called Deploy Backend to Hugging Face Spaces running
Click on it to see live logs
Green checkmark = success. Red X = something failed (see troubleshooting below)
Typical build time: 3–5 minutes (Docker build + dependency install).

Part 5 — Migration Safety
How migration works on HF Spaces
Your migrate.py is designed to be fully safe and automatic:

Every function checks if the column/table already exists before doing anything
If a column exists → skip (idempotent — safe to run 100 times)
If a column is missing → add it
The existing data in /data/tku.db is preserved
No data loss, no downtime beyond the normal restart time
For this specific fix, the migration adds supplier_id INTEGER DEFAULT NULL to products, then runs:


UPDATE products
SET supplier_id = (SELECT supplier_id FROM supplier_products ...)
WHERE supplier_id IS NULL AND EXISTS (...)
This means existing products that were already linked to suppliers (via the Suppliers page) will automatically get supplier_id set — the fix is retroactive.

What NOT to do
Do NOT manually edit /data/tku.db on HF Spaces. Do NOT delete the HF Spaces persistent storage (you will lose all data). All schema changes go through migrate.py.

Part 6 — Rollback Procedure
If the GitHub Actions deployment broke something
Option A — Quick revert (safest)


# Reverts the last commit and creates a new "undo" commit
git revert HEAD

# Enter a commit message in the editor that opens (or press Ctrl+X to accept default)

git push origin main
This creates a new commit that undoes your changes. GitHub Actions will deploy the reverted code to HF Spaces.

Option B — Go back to a specific commit


# Find the commit hash you want to go back to
git log --oneline -10

# Output example:
# def5678 Fix supplier-product relationship
# abc1234 Initial commit    ← you want to go back to this

# Reset to that commit (keeps your local changes as "unstaged")
git reset abc1234

# Or hard reset (discards local changes too — be careful)
git reset --hard abc1234

git push origin main --force
Use --force only for rollback. Never on shared branches in a team.

If the HF Space is stuck or broken
Go to https://huggingface.co/spaces/YOUR-USERNAME/tku-backend
Click Settings (top right)
Scroll to Factory reset → click Reset Space
Wait for rebuild — takes 3–5 minutes
If the problem was a bad code push, do a git revert first, then reset the Space
If the database got corrupted
Go to HF Space → Settings → Persistent Storage
You can download a backup of /data/tku.db before deleting
Delete the storage → Space rebuilds → seed data runs fresh
Note: This deletes all your live data — only do this for demo environments

Part 9 — Troubleshooting Guide
GitHub Actions failed (red X in Actions tab)
Click the failed job → expand the failed step to read the error.

Error message	Cause	Fix
remote: Invalid username or password	HF_TOKEN secret expired or missing	GitHub repo → Settings → Secrets → update HF_TOKEN
HF_SPACE is not set	Secret not configured	Add HF_SPACE secret with value YOUR-USERNAME/tku-backend
rsync: command not found	Won't happen (ubuntu-latest has rsync)	—
git push fails with 403	Token has no write access	Regenerate HF token with Write permission
HF Spaces build fails (red status)
Click Logs tab to read the Docker build output.

Error message	Cause	Fix
No matching distribution found for fastapi==X.X.X	Bad requirements.txt	Check backend/requirements.txt — verify version exists on PyPI
ModuleNotFoundError: No module named 'X'	Missing dependency	Add the package to requirements.txt
sqlite3.OperationalError in logs	Migration error	Check the migration logic in migrate.py
Space stuck on "Building" forever	Docker build hung	Settings → Factory Reset → wait for clean rebuild
Permission denied: /data	Persistent storage not mounted	Check Space settings — persistent storage should be enabled
Vercel build fails
Vercel project → Deployments → click failed deployment → read logs.

Error	Cause	Fix
Module not found: 'X'	Import path wrong	Check the import statement in the changed file
SyntaxError	JavaScript/JSX syntax error	Fix the syntax in VS Code, commit, push again
VITE_API_BASE_URL is undefined	Environment variable not set	Vercel project → Settings → Environment Variables → add it
Frontend shows old version (cache)
Force a hard refresh: Ctrl + Shift + R (Windows) or Cmd + Shift + R (Mac).

If still old: open DevTools (F12) → Network tab → check Disable cache checkbox → refresh.

Products still not appearing in Receiving after deploy
The migration might not have run yet (Space still restarting). Wait 30 seconds, refresh. If still broken:

Open HF Spaces Logs tab
Search for migrate — you should see lines like products: added column supplier_id
If you see ERROR instead, copy the error and check against migrate.py
"502 Bad Gateway" when opening the Vercel app
The HF Space went to sleep (free tier sleeps after 15 minutes of inactivity). This is normal. Wait 30 seconds for it to wake up, then refresh.

CORS error in browser console

Access to XMLHttpRequest at 'https://YOUR.hf.space/...' has been blocked by CORS policy
Fix:

HF Space → Settings → Variables
Check ALLOWED_ORIGINS — it must exactly match your Vercel URL
No trailing slash: https://tku-xyz.vercel.app ← correct
Not: https://tku-xyz.vercel.app/ ← wrong (trailing slash)
After saving, the Space restarts automatically
